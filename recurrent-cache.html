The Recurrent Cache & Slot Rollback

A different kind of cache — one slot per sequence, not one per token — and a small accommodation for speculative decoding that lets it walk backwards through time.

The KV cache is a transcript. The recurrent cache is a state machine. They share a class hierarchy and very little else — and the differences are easiest to see once you stop pretending they're alike.

In an attention-based transformer, the cache grows with the sequence: every token leaves a K and a V behind, and the next token reads all of them. In a state-space model — Mamba, RWKV, Qwen3.5 — the cache does not grow. Each layer keeps a single fixed-size state per sequence, updated in place as tokens stream through. The cache, in this regime, is not a transcript but a folded summary: a small dense object that is the sequence's past.

That changes almost everything about how the cache is bookkept. Cells are no longer per-token; they are per-sequence. find_slot does not look for a run of free positions; it looks for one slot to host an entire sequence's tail. Sequence sharing is no longer the cheap operation it was in the unified KV cache — there is no shared K/V to point at, only an opaque state vector that must be copied if a fork happens. And the most interesting addition, the one this page is mostly about, is the rollback plane: a small ring of snapshots that lets the cache un-do the last few state updates without recomputing from scratch.

scope Everything here is in src/llama-memory-recurrent.{h,cpp}. The hybrid variants (llama-memory-hybrid*) wrap this class together with a unified KV cache for models that interleave attention and recurrence; the bookkeeping below is the recurrent half.

A cell is a sequence's home

Forget what a cell meant in the unified KV cache. There, a cell was a per-token slot, and a sequence spread itself across many of them. Here a cell is a sequence — or, more precisely, a cell holds the current recurrent state for at least one sequence at one logical position. The metadata is small:

llama-memory-recurrent.h — line 88struct mem_cell {
    llama_pos pos  = -1;   // last position this state corresponds to
    int32_t  src  = -1;   // physical row to copy from on the next compute
    int32_t  src0 = -1;   // staged src for the current ubatch
    int32_t  tail = -1;   // when this cell IS a seq_id slot, the cell that owns its tail

    std::set<llama_seq_id> seq_id;  // who lives in this cell right now
};

Two fields earn their keep early. pos is the position of the last token folded into the state — not the position of a single token, but the rightmost position seen. tail is the indirection that lets a sequence-id be looked up in O(1): cell i, when i is also a valid seq_id, can be asked “where does sequence i's state currently live?” — the answer is in its tail. So a seq_id and a cell index share a number space, but the actual state for sequence i usually lives somewhere else, at cells[cells[i].tail].

This indirection is doing a small amount of clever work. It means a sequence's state can be physically moved around the cache (to make room, to gather a contiguous batch, to swap with another seq) without rewriting every reference to it; the consumer always finds the state through tail. We will see this matter in §R.3, when the allocator gathers a ubatch.

§ R.1 — tail vs. cell A seq_id n uses cell n as a name-tag, not a state container. The state lives at cells[cells[n].tail]. If you find that confusing — you're not alone; the variable name is doing double duty.

The tensor is wider than the cache

Here is where it stops looking like a normal cache. The cells array has mem_size entries — typically equal to n_seq_max, the number of concurrent sequences. But the on-device R and S tensors are taller than that:

llama-memory-recurrent.cpp — line 99const uint32_t n_rows = mem_size * (1 + n_rs_seq);
ggml_tensor * r = ggml_new_tensor_2d(ctx, type_r, hparams.n_embd_r(), n_rows);
ggml_tensor * s = ggml_new_tensor_2d(ctx, type_s, hparams.n_embd_s(), n_rows);

The metadata grid has mem_size rows; the storage tensor has mem_size · (1 + n_rs_seq). That extra dimension is a stack of snapshot planes. Plane 0 is the live state — the one the cells point at and the one the next forward pass will update. Plane 1 is yesterday's state, plane 2 is the day before, and so on, up to n_rs_seq snapshots deep.

A snapshot is not maintained explicitly. The state-update kernel writes the new state of slot s at row s of plane 0 — but the device-side update routine reads from the source row at cells[s].src0, which can point either at the live plane or, when the caller has armed a rollback, at one of the upper planes. We will spell out that addressing in §R.6. The shape of the storage tensor is the static commitment that makes the rollback possible at runtime.

find_slot does not look for runs

The allocator's job in the unified KV cache is to scan for a contiguous run of empty cells big enough to hold the next ubatch. Here, the job is gentler: one cell per sequence in the ubatch, and the contiguous-run requirement is enforced by moving cells around, not by waiting until the cache has fragments aligned. find_slot in this file is in three phases:

  • Claim. For each sequence in the ubatch, locate its current tail. If the seq does not own a cell yet, claim the next empty cell as its tail.
  • Gather. Swap whatever cells the seqs landed in so they end up at consecutive indices min .. min + n_seqs − 1. The data does not move; only the metadata is rearranged, and the indirection through tail stays consistent.
  • Stage src0 and the zero source. For each cell in the run, write src0 — the row the device should copy from on the next compute pass — and pick a free row to use as the zero state for new sequences.

The third step is where the rollback offset is folded in, but we'll get to that. First, watch the gather happen.

§ R.3 — gather The gather is what lets find_slot return a single (head, n) pair instead of a list of scattered indices. Everything downstream — the input set-up, the build_rs graph — assumes the active cells are at cells[head .. head + n).

Rollback, the short version

Speculative decoding draws a fixed batch of draft tokens, runs them through the main model in one ubatch, and then verifies how many of them were accepted. The accepted tokens are committed; the rest must be unwound. For an attention cache, unwinding is a pair of erasures: seq_rm on the unaccepted positions, done. The K and V at those positions are stale but harmless — they will be overwritten when new tokens take their place.

For a recurrent cache, there is nothing to erase. The state at position n already contains the state at position n − 1; the model folded the latter into the former. To go back to position n − 1, you have to recover the state from before the last fold. That state isn't kept by default — but with n_rs_seq > 0 it is, in one of the snapshot planes.

llama-memory-recurrent.cpp — line 180// inside seq_rm — partial rollback path
if (0 < p0 && p0 <= cell.pos && p1 > cell.pos) {
    const llama_pos rollback = cell.pos - (p0 - 1);
    if (rollback >= 1 && rollback <= (llama_pos) n_rs_seq) {
        set_rs_idx(seq_id, (uint32_t) rollback);
        cell.pos = p0 - 1;
        return true;
    }
    return false;
}

Three things in twelve lines. One: if the caller asks to delete the suffix from p0 to p1 and the cell's last position is somewhere in that suffix, this is a rollback request. Two: the depth of the rollback — how many tokens to unwind — must fit in the n_rs_seq window; otherwise the rollback is refused, and the caller will need to reseed the sequence from scratch. Three: on success, the cell's position is moved backward to p0 − 1 (the last surviving token) and a per-seq rollback index is set so the next forward pass reads from the right snapshot plane.

The state is not copied anywhere. The snapshot plane is the rolled-back state; the metadata simply chooses which plane the next compute pass reads from. The expensive part of rollback — recovering the prior state — was already paid for by writing the snapshots in the first place.

Who pays for the snapshots

The snapshots are not free. The R and S storage tensors are (1 + n_rs_seq)× larger; for Qwen3.5, the only arch in this tree that currently advertises rollback support, that is roughly the difference between a 60 MiB recurrent cache and a 240 MiB one at n_rs_seq = 3. Two consequences follow.

First, the planes have to be maintained by the model graph — when the state machine folds a new token in, it has to scroll the previous live state into snapshot 1, snapshot 1 into snapshot 2, and so on, dropping the oldest. This is not done by the cache; it is done by the model's recurrent step graph, which sees the wider storage and writes the appropriate rows. The cache is only responsible for telling the graph where to read and where to write — the addresses in src0, plus the rollback index.

Second, the rollback is per-sequence, not per-cell. A single ubatch that mixes a rolled-back sequence and a non-rolled one is fine: the rollback index is keyed by seq_id and applied when the cache emits the source row for that sequence in s_copy. We will look at that formula next.

arch gating llama-context.cpp clamps n_rs_seq to 0 unless the model is one of the architectures returning true from llm_arch_supports_rs_rollback — currently Qwen3.5 and Qwen3.5-MoE. The rest of the recurrent zoo (Mamba, RWKV, Plamo2, …) gets a normal single-plane cache.

The s_copy formula

Every compute step starts with the cache filling a tiny host-side tensor named s_copy, one entry per cell in the ubatch. Each entry is the physical row from which the device-side recurrent kernel should copy its starting state. With no rollback, the row is simply the cell's src0 in plane 0. With rollback, the cell's seq picks a plane:

return  rs_idx[seq] × size + src0;   // then reset rs_idx[seq] = 0;

Read this carefully. src0 is a slot index, 0 ≤ src0 < size. rs_idx[seq] is the chosen snapshot plane (0 for live, ≥ 1 for the past). The product walks rs_idx planes up the tensor, then adds the slot offset — a flattened 2D index into a flat row major tensor.

After the read, rs_idx[seq] is reset to zero. That is the subtle invariant: rollback is a one-shot stake. Once the kernel has read from snapshot plane k, the next compute will write the new state to plane 0 again. If you wanted to roll back the same sequence further, you would have to do it again on the next batch — but you wouldn't, because the previous rollback already consumed a snapshot, and there is nothing to roll back to.

llama-memory-recurrent.cpp — line 1244int32_t llama_memory_recurrent_context::s_copy(int i) const {
    const uint32_t cell_idx = i + mem->head;
    const int32_t  src0     = mem->cells[cell_idx].src0;

    if (mem->n_rs_seq == 0) { return src0; }

    uint32_t idx = 0;
    if (!mem->cells[cell_idx].seq_id.empty()) {
        const llama_seq_id seq = *mem->cells[cell_idx].seq_id.begin();
        if (seq >= 0 && (size_t) seq < mem->rs_idx.size()) {
            idx = mem->rs_idx[seq];
            mem->rs_idx[seq] = 0;   // one-shot
        }
    }
    return (int32_t)(idx * mem->size) + src0;
}
Invariant

A snapshot is a read-only state for exactly one upcoming compute. The cache trusts the model graph to maintain the snapshot stack after each accepted step. There is no reference counting on the planes; if rollback is requested but the snapshot wasn't kept, the kernel reads stale or corrupt state.

The zero plane (rs_z)

One more wrinkle. A new sequence has no prior state. When find_slot lays out the ubatch, it scans the active cells looking for one whose src is unset — never been written, no prior reference — and tags it as rs_z. The model's recurrent step uses this row as its zero source: any cell in the ubatch whose src < 0 (a new sequence) copies from rs_z, which the backend has guaranteed to be all zeros. It is not really a separate plane; it is a slot in plane 0 that happens to hold the canonical zero state for the duration of this batch.

If you wonder why this is needed instead of just emitting a src0 = -1 sentinel and letting the kernel branch: state-space kernels are unrolled across the whole batch, and a branch-free copy from a known-zero address is faster than a per-cell predicate. The cost of holding one slot's worth of zero on the device is negligible compared to the kernel saved by the assumption.

§ R.7 — zero plane rs_z = -1 when the entire active run already has prior states; find_slot still constructs it, but no cell needs it. It is asserted non-negative only when at least one cell asks for the zero source.

Why splits become sequential

Tucked away in init_batch, with a TAG comment, is a small but consequential restriction. When n_rs_seq > 0 the batch is split by sequence rather than equally:

llama-memory-recurrent.cpp — line 419if (n_rs_seq > 0) {
    // [TAG_RECURRENT_ROLLBACK_SPLITS]
    // TODO: recurrent state rollback does not support equal splits
    ubatch = balloc.split_seq(n_ubatch);
} else {
    ubatch = balloc.split_equal(n_ubatch, true);
}

The reason is a property of the snapshot stack. An equal split groups tokens by position across many sequences and feeds them in lock-step; that would require the snapshot plane for sequence A and the live plane for sequence B in the same kernel invocation, indexed independently per-row. The current state-update kernels assume a single rollback index per kernel launch — applied uniformly, after the metadata gather has placed all the active cells at head .. head + n. Sequential splits route each sequence's tokens through their own kernel pass, which is where the per-seq rollback index can be applied cleanly.

This is a trade-off the implementation made on purpose. Equal splits would extract more parallelism from large batches; sequential splits make the rollback path tractable. For the speculative-decoding workload that motivated rollback — a handful of sequences with short drafts — sequential is the right default.

State save / load with a pending rollback

Two final places the planes show up: the serialization paths. When state_write exports a sequence with a pending rollback, the row it streams out is not the live plane's row; it is the snapshot plane's row corresponding to the current rs_idx[seq]. The formula is the same:

llama-memory-recurrent.cpp — line 772const uint32_t cell_id = rs_idx_cur * size + (cell.src >= 0 ? cell.src : (int32_t) i);

The logical state for sequence s at the time of the save is the snapshot. If you saved the live plane, you would be saving the state from before the rollback was applied — which the rollback was specifically meant to abandon. So the writer reaches into the chosen plane and exports those rows instead, and the reader, on load, drops the rs_idx back to zero: the loaded plane is the new live state, with no rollback history.

§ R.9 — shared seqs A constraint hides here: if a cell is shared between multiple sequences with different rollback indices, the writer aborts with cannot write shared recurrent state with different rollback indices. The serialization format cannot express "this row is two different states depending on who is reading it."

A summary

The recurrent cache is a small, opinionated allocator over a state vector. Most of its bookkeeping is unsurprising — one slot per sequence, a tail indirection for fast lookup, a gather step to keep ubatches contiguous. The rollback machinery is the addition that distinguishes it: a stack of snapshot planes, sized at construction, addressed by a per-sequence index, with a one-shot read semantics that aligns with the way the model graph maintains the stack. The whole apparatus is justified by exactly one workload — speculative decoding against a state-space model — and the design choices reflect that focus: per-seq, not per-cell; sequential splits, not equal; gated by architecture, not on by default. It is a useful illustration of how a cache earns its complexity only when there is an existing user that needs precisely that complexity, and not before.

If you want to compare it against the attention-based cache, the contrast is clean: the unified KV cache earns its bookkeeping by storing many tokens and reusing prefixes; the recurrent cache earns its bookkeeping by storing one summary and being able to step that summary backward. Different operations, different invariants, the same class hierarchy, and a remarkable amount of shared scaffolding in llama-memory.h binding them together.
