Gemma 4 MTP — how the assistant shares the trunk KV cache

trunk: 60 layers assistant: 4 MTP blocks one KV, two contexts The assistant draft never owns K/V. Its Q-only blocks read trunk K/V views directly each decode.

1. The mapping: assistant block → trunk source layer

Gemma 4 31B has 64 layers: trunk 0..59 and MTP assistant 60..63. Each assistant block computes only Q (with its own q_proj and q_norm) and reads K/V from a fixed trunk layer chosen by attention type. SWA blocks borrow from the trunk's last SWA layer (58); the full-attention block borrows from the trunk's last full layer (59).

trunk layer (owns K/V) MTP block (Q only) SWA attention (window=1024) full attention K/V read

SWA assistant blocks all share trunk L58's K/V; the single full block borrows L59. Each assistant Q has its own head dim (256 for SWA, 512 for full) which matches the trunk source layer it reads from.

2. Two contexts, one model, one KV cache

Trunk and assistant weights live on the same llama_model. Two llama_contexts wrap it: ctx_tgt (LLAMA_CONTEXT_TYPE_MAIN) decodes the trunk and owns the KV. ctx_dft (LLAMA_CONTEXT_TYPE_MTP) decodes the assistant draft graph. A KV filter excludes layers 60..63 from owning storage in ctx_dft, so there is exactly one allocation per K/V tensor.

src/llama-context.cpp:1745 snapshots the trunk memory at the top of ctx_dft->decode(); the snapshot is dropped via RAII at function exit. No external setter API or long-lived sharing object.

3. Per-decode snapshot lifecycle

Trunk K/V buffers are allocated by the trunk's KV cache and live for the lifetime of the model run. The MTP scheduler's tensor router (ggml_backend_sched_backend_from_buffer) picks a backend by looking up the trunk buffer's buft — since both contexts share model.devices, the assistant scheduler automatically supports the trunk-KV buffer type. The view tensors produced by get_k / get_v stay valid across scheduler resets.

The snapshot is just a memory-context handle (llama_memory_context_ptr), not a copy. It's plumbed into llm_graph_params.src_mctx and consumed by the custom input class.

4. Inside the assistant attention call

Existing input classes (llm_graph_input_attn_kv) assume the graph's own mctx is a KV cache. For ctx_dft that's empty, so we use llm_graph_input_attn_src_kv_iswa which holds a pointer to the trunk's iSWA memory context. The mask is built over the trunk's physical n_kv, and only logical accepted positions are unmasked. SWA layers use the SWA window from hparams.n_swa.

Mask construction

// trunk's last accepted position
p = accepted_len - 1

for each stream s, query r, kv slot k:
  pos = src_mctx->seq_pos_of(k, s)
  full_ok = pos < accepted_len
  swa_ok  = pos > p - hparams.n_swa
  mask[s,0,r,k] = (full_ok && swa_ok)
                ? 0.0f : -INFINITY

Within one AR pass the trunk's accepted prefix is fixed, so the mask is identical across all 6 draft steps — that's what makes graph reuse worth doing.

5. Multi-stream slicing (kv_unified = false)

When the trunk runs with kv_unified=false, its K/V tensors carry an extra stream dimension k->ne[3] = n_stream. The assistant ubatch only references the streams of currently active draft sequences. Without slicing, build_attn_mha would split Q across all trunk streams and collapse tokens. The fix takes a contiguous range view of K/V over the streams referenced this decode.

6. End-to-end speculative loop

Unlike Qwen3.5 MTP — where the draft block computes its own K/V and needs a catch-up decode — Gemma 4's assistant has no KV to keep in sync. That step is dropped. The only state crossing the trunk/assistant boundary is the recurrent hidden pending_h (5376-dim) plus the last accepted token id.

The trunk verify in step 4 mutates trunk KV — that's why the snapshot must drop before it runs. RAII inside ctx_dft->decode() (see src_mctx_reset_on_exit at llama-context.cpp:1738) guarantees it.
