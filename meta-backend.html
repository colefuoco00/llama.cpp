The Meta Backend

A virtual device that wraps a fleet of real devices, hides the sharding from the scheduler, and decides when the partial sums on each card need to meet.

Tensor parallelism, in ggml, is not a runtime feature you opt into. It is a backend you ask the scheduler to use, and the trick is that the backend appears to be a single device while owning a row of them.

The meta backend is ggml's answer to multi-GPU. From the outside it presents the ordinary backend interface: a device, a buffer type, a buffer, a graph_compute. From the inside, each of those is a thin shell wrapping a vector of simple backends — one per physical device — together with the bookkeeping that decides, for every tensor flowing through the graph, how it is split across them. A mul_mat node fans out into N parallel mul_mats on N devices, each operating on a slice; the slices are then either disjoint pieces of the output, or partial sums that an AllReduce will fold together. The scheduler does not see any of this. It hands the meta backend a graph and gets a result.

This page walks the seams of that abstraction. There are four pieces to know: the split state that classifies every tensor's distribution across devices; the per-op handlers that derive output splits from input splits; the graph-compute driver that breaks the work into subgraphs at AllReduce boundaries; and the reduction itself, which is either backend-native (NCCL) or a butterfly fallback woven out of ordinary ggml graph nodes.

scope Everything here is in one file: ggml/src/ggml-backend-meta.cpp (~2,100 lines). The public surface is in ggml-backend.h: a single device-creation entry point, an enum of split axes, and a callback the caller supplies to assign split states for parameter tensors.

A device that is several devices

The construction is a curried call. You hand the meta backend a list of simple devices — k CUDA cards, two Metal devices, a CPU plus a Vulkan card, whatever the registry offers — together with a callback that will be invoked, later, for each parameter tensor to ask how it should be split:

The returned dev behaves like any other device: it advertises a name (the concatenation of its members), an aggregate memory budget (the sum), a buffer type (a meta buffer type that delegates to each simple buffer type underneath), and a set of capabilities computed as the intersection of its members'. From the scheduler's point of view, the meta device is a single device and the only one it sees.

The interesting work happens when a tensor is allocated. The meta buffer holds N simple buffers (one per device, each backed by that device's allocator) and a map from ggml_tensor * to a vector of N simple tensors. The simple tensors are real, allocated, address-bearing — one per device, dimensioned according to that device's slice. The high-level tensor that the rest of ggml sees is a wrapper that resolves into its j-th physical incarnation when a backend operation needs to touch the data on device j.

§ M.1 — buffer math A meta buffer's alignment is the LCM of its members' alignments (asserted to actually be a common multiple). Its max_size is the minimum across members. Its alloc_size per tensor is the max — overcommitted, because not every slice has the same shape after the split.

  • AXIS_0..3 — the tensor is split along that ggml dimension. Each device owns a contiguous slice. For most tensors this is one segment; QKV-fused matrices use multiple segments so that each device gets a piece of Q, K, and V.
  • MIRRORED — every device holds the full tensor. Used for activations that are cheap enough to broadcast, and for small parameters (norms, biases).
  • PARTIAL — each device holds an addend; the logical tensor is their sum. This is the state immediately after a mul_mat that reduces along the split axis, and the only state that requires an AllReduce to be made consumable again.
  • NONE / UNKNOWN — internal sentinels used by the inference logic to signal "no decision yet" and "contradiction"; never the final state of a tensor.
§ M.2 — segments A tensor can have more than one segment when its layout is inhomogeneous along the split axis — fused QKV is the canonical example. Each segment carries its own per-device ne array, so device j ends up with a slice of Q, a slice of K, and a slice of V at the right offsets.

Inferring the split, op by op

Parameter tensors have their split assigned externally — the caller supplies a function that, given the tensor, returns a split state. Subgraphs, cut at the reductions

Now the graph. Once every node has a split state, the driver walks the linearized cgraph and partitions it into subgraphs — runs of nodes that can execute on each device independently, separated by AllReduce points. The cut comes at exactly the points where a node's split state is PARTIAL: each device finished its share of the work, and the next op cannot proceed until the partials meet.

Each subgraph is then materialised N times — one copy per simple backend, with the high-level nodes replaced by their per-device simple tensors. The driver calls ggml_backend_graph_compute_async on each device's subgraph in turn, then runs an AllReduce on the boundary, then proceeds to the next subgraph.

The subgraph plan is cached on the input cgraph's UID. If the same graph comes through twice — which is the common case in a streaming decode — the second pass skips the partition step entirely and just dispatches the prebuilt per-device subgraphs.

§ M.4 — assume_sync The mul_mat handler takes an assume_sync bool. When the caller (set_tensor, get_tensor) just wants to know "where is this tensor's data logically", an AXIS_0 × AXIS_0 mul_mat is reported as MIRRORED — because by the time the data is read, the reduction will have happened. The compute path uses assume_sync=false and gets back PARTIAL so it knows to insert the reduction.

The AllReduce, two ways

When the partition is done, what remains is to take a set of PARTIAL tensors — one on each device — and reduce them so every device ends up with the sum. The meta backend tries first to ask the underlying backend whether it has a native collective; CUDA, for example, exports ggml_backend_comm_allreduce_tensor backed by NCCL. If that returns true, the reduction is delegated and the driver moves on. If not, the fallback builds the reduction out of ordinary ggml nodes.

The fallback is a butterfly: at each step k, device j exchanges its tensor with device j XOR 2k. The exchanged tensor is copied through an auxiliary buffer (one per step, sized to the largest PARTIAL in the graph), then summed into the local tensor via an OP_ADD node. After log₂N steps every device holds the full sum.

Two details from the implementation are worth surfacing. First, the auxiliary buffer rotates per step (i_buf in the code), so a slow device does not stall the next step's data motion: while it is still summing, the next step's incoming tensor lands in a different buffer. Second, when N is not a power of two, the excess devices first push their partial into one of the lower indices, then participate by sitting out the butterfly and receiving the final sum at the end — keeping the inner loop a clean log₂ of the round number.

§ M.5 — when there is no comm On a single GPU the meta backend is overhead with no benefit; the comm_init is skipped and the AllReduce path is never taken. The split states still get computed, but every tensor ends up MIRRORED and there is one subgraph for the whole compute.

Delaying the reduce, for MoE

Mixture-of-experts layers produce a PARTIAL tensor that is immediately followed by a small constellation of MIRRORED operations: a scale by gate weights, possibly an add of two expert outputs, sometimes a final mul by a router score. Each of those ops can be done locally on the partial sum, because their MIRRORED other operand is the same on every device. The reduction can wait until after the constellation, when there is exactly one tensor to AllReduce instead of two or three.

The driver looks for this pattern with a helper named get_i_delayed. Given a node index i that would have ended the subgraph, it scans forward across MIRRORED ops whose operands all share a split with the candidate — first an optional ADD_ID for the MoE bias, then a chain of MULs, then a final ADD that combines per-expert contributions. When the pattern fits, i is advanced to the end of the chain; the subgraph swallows the constellation and the reduction happens once, at the chain's tail.

There is a wrinkle. If a particular device's slice is zero-sized — for example, a gating tensor that has all-zero rows on this rank — the participating node must have its GGML_TENSOR_FLAG_COMPUTE cleared so that the simple backend does not attempt to operate on an empty tensor. The delay logic propagates that clear flag forward through the chain, so the eventual AllReduce sees zeros from that device rather than uninitialised memory.

What this buys, and what it costs

The shape of the abstraction is doing several things at once. First, it makes multi-device a backend concern, not a scheduler concern: the rest of ggml — the operator library, the cgraph machinery, the public C API — is none the wiser. Second, it makes the sharding declarative: the caller assigns split states to parameters once, and the inference rules carry that decision through every op without further configuration. Third, it makes the collective costs visible in the graph itself: an AllReduce is a subgraph boundary, and the boundaries are countable from the partitioner's output. You can read the cost of a multi-GPU forward pass off the cgraph the same way you read the cost of a single-GPU one.

The price is paid in coverage. Every new op needs a split-state inference rule (see the table in §M.3); ops that lack one will hit a GGML_ABORT the first time they appear in a meta-backend graph. The reduction is communication-bound, and the butterfly fallback assumes peer-to-peer copy between simple backends works — true on multi-GPU CUDA, less so when the simple backends are heterogeneous. The MoE delay is a peephole that works only for the specific MUL/ADD/ADD_ID shape; other constellations would each need their own heuristic.

Comparison

The meta backend is one of three multi-device strategies in ggml. The older layer-split mode partitions whole layers across devices and is pipeline-parallel (each device runs different layers serially); the older row-split mode operates inside a single mul_mat. The meta backend generalises both: row-split becomes one possible per-tensor decision, and pipeline overlap falls out of the fact that subgraphs on different devices can run asynchronously between AllReduce barriers.

A summary

The meta backend is a single device that owns several. It hides the sharding behind the ordinary backend interface, classifies every tensor by how it is distributed (five split states), derives output splits from input splits one op at a time, partitions the cgraph at the points where a reduction is needed, and runs an AllReduce — backend-native if it can, butterfly-by-graph-nodes otherwise — to bridge across the cuts. There is a peephole that delays the reduce past mirrored ops to make MoE tractable, and a cache keyed on cgraph UID to amortise the partitioning across repeated forwards.

Compared to the recurrent cache or the KV cache, this is a much more horizontal kind of complexity: the file is long not because the ideas are deep but because each ggml op needs its own inference rule, and there are a lot of ops. The interesting kernel is small. The interesting question — “how should this tensor be sharded?” — is answered op by op, in twenty-line handlers, and the rest of the machinery follows from those answers like consequences from a definition.
