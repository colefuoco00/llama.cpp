Storage

The Layout in Memory

Where the K and V actually live: their shapes, their dtypes, the curious matter of whether V is stored transposed, and what a quantized cache buys.

A tensor is a shape and a dtype. Once you know both, the rest is allocator arithmetic — but the shape is the thing to get right first.

The cache is not one tensor. It is two tensors per layer, repeated L times. The constructor in src/llama-kv-cache.cpp walks the model's layer list, asks the model what dimensions K and V should have at that depth, and allocates a pair of buffers. The result lives in a vector of kv_layer structs, each holding a K tensor, a V tensor, and a handful of pre-computed views into them.

src/llama-kv-cache.h — line ~215struct kv_layer {
  // the layer index in the model (a layer may be skipped, e.g. MTP)
  uint32_t il;

  ggml_tensor * k;          // [n_embd_k_gqa, kv_size, n_stream]
  ggml_tensor * v;          // [n_embd_v_gqa, kv_size, n_stream]   (or transposed)

  std::vector<ggml_tensor *> k_stream;  // per-stream views
  std::vector<ggml_tensor *> v_stream;
};

The shape, piece by piece

The K tensor has three dimensions. The first, n_embd_k_gqa, is the total width of the key projection — the per-head dimension multiplied by the number of key heads. In a vanilla multi-head transformer the number of key heads equals the number of query heads; under grouped-query attention (GQA), it can be much smaller. The GQA in the name is the giveaway: this is the after-grouping width, the one that actually has to be stored.

§ 2.1 — gqa Grouped-query attention is one of the most important savings in modern LLM design precisely because it shrinks this dimension. Eight query heads sharing one key head means the cache pays for one head, not eight.

The second dimension, kv_size, is the capacity of the cache in tokens — how many slots it has. This is fixed at construction. The third dimension, n_stream, is the number of physical partitions, which we will not dwell on here because Chapter IV is devoted to it. For now, treat it as 1 in your head.

The V tensor has the same shape on paper — substitute n_embd_v_gqa for the first dim — but a switch in the constructor controls whether the storage is transposed:

src/llama-kv-cache.cpp — line ~200// when v_trans is true, V is stored as [kv_size, n_embd_v_gqa, n_stream]
// otherwise as [n_embd_v_gqa, kv_size, n_stream], like K.

if (v_trans) {
    layer.v = ggml_new_tensor_3d(ctx, type_v, kv_size, n_embd_v_gqa, n_stream);
} else {
    layer.v = ggml_new_tensor_3d(ctx, type_v, n_embd_v_gqa, kv_size, n_stream);
}

Transposing V is not for elegance; it is for the matmul. The attention output for a single query is softmax(Q·Kᵀ) · V. When V is stored transposed (kv_size along the contiguous axis), the right multiplication of the softmax vector with V becomes a stride-friendly contraction. Whether to do it depends on the backend and on whether flash attention is in use; the cache supports both because the surrounding compute graphs do.

A picture of one layer

The dtype question

K and V do not have to be stored in the same dtype the model computes them in. The constructor accepts a type_k and type_v independently; you can store K at fp16 and V at q8_0, or both at q4_0, or any other combination GGML's quantization machinery supports. There is no special path for quantized cache reads or writes: the tensors are allocated with the requested type, and the matmul kernel does the dequantization on the fly, the same way it does for the model's frozen weights.

This is one of those moments where the simplicity of the GGML abstractions pays off twice. Because every operation that touches K or V goes through ggml_mul_mat (or an equivalent), the choice of dtype is a property of the tensor, not of the cache logic. The seq_cp, seq_rm, find_slot machinery never touches the data — it only touches metadata. So as far as the cache logic is concerned, the bytes in the tensor are opaque.

Operating point

At 32k context, a 7B-class model with n_kv_heads = 8 and head_dim = 128 across 32 layers stores roughly:

  • fp16: ~4.0 GiB total cache
  • q8_0: ~2.1 GiB total cache
  • q4_0: ~1.1 GiB total cache

The quality cost of q8_0 is, in practice, negligible. The quality cost of q4_0 is real and depends on the model.

What gets allocated, and when

All cache memory is allocated up front, at the moment the llama_kv_cache is constructed. There is no dynamic growth: if the cache fills, the eviction logic — or, more commonly, the caller's batching code — must make room. The motivation is partly performance (no allocator calls in the hot path) and partly platform (GPU buffers do not like to grow). The cost is that kv_size has to be chosen at the start of the session, and chosen large enough.

src/llama-kv-cache.cpp — constructorfor (uint32_t il = 0; il < n_layer_cache; ++il) {
    const uint32_t n_embd_k_gqa = hparams.n_embd_k_gqa(il);
    const uint32_t n_embd_v_gqa = hparams.n_embd_v_gqa(il);

    ggml_tensor * k = ggml_new_tensor_3d(ctx_layer,
                          type_k, n_embd_k_gqa, kv_size, n_stream);
    ggml_tensor * v = ggml_new_tensor_3d(ctx_layer,
                          type_v,
                          v_trans ? kv_size       : n_embd_v_gqa,
                          v_trans ? n_embd_v_gqa  : kv_size,
                          n_stream);

    layers.push_back({ il, k, v, /* per-stream views */ });
}

Note n_layer_cache rather than n_layer. Some architectures (multi-token-prediction heads, certain hybrids) do not cache every layer. The cache walks only the layers the model actually wants cached, and kv_layer::il records the original layer index so the rest of the graph can find the right buffers.

cf. § 8.2 Gemma 4's multi-token-prediction (MTP) heads are an example: the draft heads share the trunk's K and V for the layers they re-use, and only cache K/V for the layers they introduce.

Variable V-width: the padding trick

For most models, every layer's V embedding has the same width, and the cache is a simple stack of identically-shaped tensors. For a few architectures, V widths differ between layers — and if flash attention is not available to handle this directly, the surrounding compute graph wants the V tensors to have constant shape so it can be re-used across decode steps. The cache handles this by allocating every layer's V to the maximum width found in the model, and recording the imbalance as a flag (the implementation labels this TAG_V_CACHE_VARIABLE). The unused entries are simply written and ignored.

It is a slightly wasteful arrangement, and it exists only for the awkward intersection of variable V widths and no flash attention. When flash attention is available, the kernel can consume the layer-specific widths directly and no padding is needed.

What we have not stored

It is worth saying explicitly what the cache does not contain. It does not store the queries (queries are computed once per step and immediately consumed). It does not store the hidden states (the model state at each position is not needed once K and V are derived from it). It does not store the attention outputs. It does not store the RoPE-rotated form of K — RoPE is, in this implementation, applied on the way in to K so that what is cached is the rotated form, and re-rotation under position shifts is handled by a special graph (see Chapter VII).

What is in the cache is exactly what attention needs to consume: post-projection, post-rotation K and V vectors, indexed by physical slot. Everything else is metadata, and that metadata is the subject of the next chapter.
