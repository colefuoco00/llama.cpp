The Lifecycle of an Operation

From find_slot through apply_ubatch to update. And the seq_* family that manipulates the cache between forward passes.

The cache has a hot path and a cool path. The hot path runs every decode step. The cool path runs only when sequences are created, forked, trimmed, or shifted. Both are short. Both are interesting.

A forward pass through the model consumes a ubatch — a small batch of tokens, each labelled with a position and a sequence id — and produces logits. From the cache's point of view, the work is to find a home for the new K and V vectors the model is about to compute, to write them there, and to update the metadata accordingly. This is the hot path. It is three function calls long.

The three-step waltz

conceptual — one decode step// 1. find homes for the new tokens
slot_info slots = cache->find_slot(ubatch);

// 2. commit the metadata: positions, sequences, shifts.
cache->apply_ubatch(slots, ubatch);

// 3. run the model. it writes K, V into the slots and reads the mask.
run_graph(model, ubatch, cache);

// 4. apply deferred work (k-shifts, stream copies, defrag).
cache->update();

Step 1 is what we described in Chapter III: a scan over the cells for a viable run. Step 2 writes pos, seq, and shift for each slot the ubatch will fill, but does not touch the K and V tensors. Step 3 is the model running, which writes the actual K and V into the slots step 2 announced. Step 4 catches up on the deferred work — re-rotating K to absorb position shifts, copying data between streams when a previous seq_cp demanded it, and so on.

The split between steps 2 and 3 is deliberate. The cache cannot write K and V — only the model can, because only the model knows what they are. But the cache can commit to where they will go before the model runs, and that lets the surrounding code build the rest of the forward-pass graph with concrete tensor offsets baked in. The model's graph reads from a slot, writes to a slot, and the cache has the slot numbers ready.

prepare, the dry run

For multi-ubatch workloads, the cache offers a prepare step that simulates several ubatches in advance. The motivation: when a large batch is split into N smaller ubatches by the batch driver, the cache needs to know that all N will fit before committing to the first one. A naive approach — find_slot for each ubatch in sequence, see if any fail — would mutate the metadata as it went, and would have to be rolled back on failure.

src/llama-kv-cache.cpp — line ~676std::vector<slot_info> prepare(const std::vector<ubatch> & ubatches) {
    // snapshot the metadata grid
    auto snapshot = cells.save();

    std::vector<slot_info> results;
    for (const auto & ub : ubatches) {
        slot_info s = find_slot(ub);
        if (s.is_error()) {
            cells.restore(snapshot);   // roll back everything
            return {};
        }
        apply_ubatch(s, ub);          // temporarily — for the next find_slot
        results.push_back(s);
    }

    cells.restore(snapshot);          // undo the simulation
    return results;
}

The trick: simulate to find the slots, snapshot to undo the simulation, return the results, and let the caller commit the ubatches one by one for real. Because the metadata grid is short — a few arrays of length kv_size — saving and restoring it is cheap. Because the data tensors are never touched in this simulation, the gigabytes of cached K and V never move.

The seq_* family

Between forward passes, the caller may manipulate sequences. The most useful operations form a small family. Each is a metadata-only operation in the unified-stream case; the per-stream case may queue a deferred data copy, as discussed in Chapter IV.

OperationWhat it doesTouches data?
seq_rm(s, p0, p1) remove sequence s's ownership of cells in position range [p0, p1); cells with no remaining owners become empty. no
seq_cp(s_src, s_dst, p0, p1) add s_dst to the bitset of every cell in [p0, p1) that belongs to s_src. Fork in O(k). only if cross-stream
seq_keep(s) remove every sequence except s from every cell. Used to discard speculative branches. no
seq_add(s, p0, p1, delta) shift the cached positions of sequence s in [p0, p1) by delta. Used for sliding context windows. only K, re-rotation deferred
seq_div(s, p0, p1, d) divide cached positions by d. Used for context compression. only K, re-rotation deferred
clear() empty every cell. no

The two operations that do touch data — seq_add and seq_div — touch only K, and they touch it lazily. Each cell records the shift it has accumulated in the shift field; the next update() runs a graph that re-rotates the K of every cell with a non-zero shift, in place, on the device. The host never sees the rotated bytes. The V tensor needs no shift because V is invariant under position changes — it is K and Q that RoPE rotates.

§ 7.3 — invariant Every seq_* operation that returns successfully leaves the cache in a state where the next find_slot will see consistent metadata. Failure modes (out of slots, invalid range) leave the cache untouched.

Defrag without copying

A cache that has churned through many sequences may end up with its non-empty cells scattered through the slot space, with no contiguous run large enough for the next ubatch. The classical answer is defragmentation: move cells together to consolidate the free space.

The cache offers a defrag operation, but it is unusual. It does not move K and V. Instead, it re-numbers the metadata — assigns new slot indices to the existing tokens, in a packed layout — and emits a graph that permutes the K and V tensors on the device to match the new numbering. The host CPU does the bookkeeping; the device does the data motion. The bytes never cross the bus.

conceptual — defrag// 1. build a permutation that packs the used cells to the front
std::vector<uint32_t> perm = build_pack_permutation(cells);

// 2. emit a graph that applies the permutation to every layer's K and V
for (auto & layer : layers) {
    graph.add_permute(layer.k, perm);
    graph.add_permute(layer.v, perm);
}

// 3. update the metadata to reflect the new layout
cells.apply_permutation(perm);

The same trick — let the device do the data motion, let the host do the bookkeeping — appears throughout the cache. Position shifts: shift the metadata, defer the K rotation to a device-side graph. Stream copies: queue them, run them in update. The cache, viewed this way, is a careful split between two kinds of state: a small, cheap, host-resident set of metadata that can be manipulated freely, and a large, expensive, device-resident set of data that is only ever transformed by graphs.

apply_ubatch and the overlap rule

One subtle case in step 2 of the waltz is worth examining. When the cache commits a ubatch into a slot range, what happens if some of the slots in that range were not empty — if, for example, the SWA-aware allocator chose to reclaim cells that had fallen out of the window? The answer is that apply_ubatch overwrites them: the previous owners (who had been masked out by SWA anyway) are evicted, the new positions and sequences are written, and the cells become the new ubatch's. The data tensors will be overwritten in step 3 by the model.

src/llama-kv-cache.cpp — line ~1017void apply_ubatch(const slot_info & s, const ubatch & ub) {
    for (uint32_t i = 0; i < ub.n_tokens; ++i) {
        const uint32_t slot = s.start + i;

        // if the slot was previously occupied, drop the old owner.
        if (cells.pos[slot] != -1) {
            cells.evict(slot);
        }

        cells.pos[slot]    = ub.pos[i];
        cells.seq[slot]    = ub.seq_id[i];
        cells.shift[slot]  = 0;
        cells.mark_used(slot);
    }
}

The eviction is, again, metadata-only. The cell's old K and V will be overwritten by the model's graph in step 3; until then, the bytes are stale but invisible — the mask will treat the slot as belonging to the new sequence at the new position.

What runs in update

The final step, update(), is the cache's cleanup pass. It handles:

  • k-shifts — re-rotate K for cells whose shift field is non-zero, then clear those shifts.
  • stream copies — execute any sc_info entries that seq_cp queued, copying K and V from source stream cells to destination stream cells via ggml_backend_tensor_copy.
  • defrag — if requested, run the permutation graph described above.

If none of those are pending, update is a no-op. In the steady-state decoding of a single sequence, it usually is.

Summary of the dance

find_slot → apply_ubatch → run model → update. Repeat. The first two are metadata work; the third is the model's. The fourth is deferred device work. The hot path of a 200-token-per-second decode loop spends almost all its time in step 3 and is essentially free in 1, 2, and 4.
