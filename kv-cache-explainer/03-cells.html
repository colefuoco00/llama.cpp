Bookkeeping

Cells, Slots, and the Allocator

Where the cache stops being a tensor and starts being a bookkeeping problem. The metadata grid that decides which token lives where.

A cache that knows what it stores is worth several caches that don't. Most of the cleverness in llama.cpp's KV cache is in the metadata.

In the last chapter we described the cache's data — the K and V tensors, allocated up front, indexed by physical slot. Now we turn to its metadata: the bookkeeping that says which logical token (sequence X, position Y) is stored in which physical slot. This is the work of the class llama_kv_cells_unified, defined in src/llama-kv-cells.h. It is the most opinionated part of the cache, and the part that earns its name.

A cell is a record

A cell, in this code, is the per-slot bundle of metadata. The class stores four parallel arrays of length kv_size — one entry per physical slot. For slot i:

src/llama-kv-cells.h — line ~488class llama_kv_cells {
    std::vector<llama_pos>      pos;    // position in the sequence, -1 if empty
    std::vector<llama_pos>      shift;  // accumulated position shift (RoPE-relevant)
    std::vector<seq_set_t>      seq;    // bitset: which sequences own this slot
    std::vector<llama_kv_cell_ext> ext; // 2D coords for M-RoPE

    // derived: the set of cells that are non-empty,
    // kept up to date for fast iteration.
    std::set<uint32_t> used;
};

The four fields together answer four questions:

  • pos — at what logical position does this slot live? A sentinel of -1 means the slot is empty.
  • seq — which sequences claim it? A bitset, because a single slot can be owned by several sequences simultaneously (more on this in §3.4).
  • shift — has the position been displaced from its original value? RoPE cares; the rest of the cache mostly does not.
  • ext — for models that use 2D positions (M-RoPE, multimodal), what row and column does this slot represent?
§ 3.1 — design The split between data and metadata is what makes seq_cp, seq_rm, find_slot, and friends cheap: they manipulate a few short arrays, not the gigabytes of cached K/V.

And one derived value: used, the set of slot indices that are currently non-empty. It is kept in sync with the pos array as slots are claimed and released. The set lets the mask-builder, the defrag logic, and the various seq_* operations iterate only over the cells that matter — usually a tiny fraction of kv_size.

The real algorithm is a little more careful than the toy. It considers SWA-masked cells as candidates for reclamation, it prefers runs that have been free the longest, and it understands sequence sharing — but the kernel of it is what you see: a scan for a contiguous run of free slots large enough to hold the incoming ubatch.

find_slot, in earnest

The allocator entry point is find_slot, in src/llama-kv-cache.cpp. Its job: given a ubatch of m tokens (each with a position and a sequence-id), find a set of physical slots large enough to hold them, satisfying the constraints of the cache configuration. In the default unified mode, this means a contiguous run; in the per-sequence mode it means a run within a specific stream. SWA-aware caches additionally treat cells that have fallen out of the sliding window as reclaimable.

src/llama-kv-cache.cpp — ~ line 818slot_info find_slot(const llama_ubatch & ubatch, ...) {
    const uint32_t n_tokens = ubatch.n_tokens;

    // scan the cells array looking for a viable home for this ubatch.
    // "viable" depends on:
    //   - the stream the sequences map to
    //   - which cells are empty (pos == -1)
    //   - which cells are masked out by SWA and can be reused
    //   - whether we need a contiguous run (unified) or can scatter

    for (uint32_t s = 0; s + n_tokens <= kv_size; ++s) {
        if (can_place_at(s, ubatch)) {
            return slot_info{ /* starting slot s */ };
        }
    }
    return slot_info::error();
}

If find_slot returns an error, the batch driver has options. It can try a smaller ubatch. It can request a defrag — though, as we'll see in Chapter VII, defrag in this cache is mostly accomplished without moving data. Or it can give up and report the failure upward.

Sequence sharing

Here is the move that makes the cache more than a buffer. A single cell can be owned by more than one sequence, as long as they agree on what is stored there. The seq field is a bitset, not a single index. When two sequences share a prefix — most commonly because one was forked from the other — they share the cells, too. Forking a sequence is therefore an O(k) operation in metadata only, where k is the prefix length, and a no-op in data.

Sequence sharing is what makes speculative decoding tolerable. A draft model produces several candidate continuations from the same prefix; if each candidate had to recompute or duplicate the prefix's K and V, the speculation would cost more than it saved. With shared cells, the draft sequences fork off the trunk for free and pay only for their unique tails.

Position shift

The shift field is a small concession to a model architecture decision. RoPE rotates K (and Q) by an angle proportional to position; once a K has been rotated and stored, you cannot interpret it as belonging to a different position without un-rotating and re-rotating it. So when a sequence's positions are shifted — typically by seq_add, used to slide a context window — the cache cannot just change pos and call it a day. It must, eventually, re-rotate the stored K to the new positions.

It does this lazily. The shift field records that a slot's position has been displaced by some delta but its K has not yet been re-rotated. A separate compute graph, run on the next forward pass, walks the cells with non-zero shifts and applies the rotation update on the device, without ever pulling the K back to the host. The shift is then cleared.

cf. § 7.4 The deferred K-shift graph is one of the more clever bits of engineering in the cache: defrag and shift are both done without data motion outside of the GPU.

2D positions

The ext field is, for most models, dead weight. It exists for those that need it — M-RoPE-style models that index tokens not by a single integer but by a pair (row, column), and so need to record both for each cell. The cache happily carries the extra information through; the mask-builder consults it when constructing 2D-aware attention. For a vanilla 1-D model, ext is uninitialized and ignored.

A summary, before we move on

The metadata grid is small, dense, and almost free to query. Every operation on the cache that does not touch data — and that is most of them — operates on it. The grid is what makes a fork O(k), a sliding window an O(1) mask predicate, and a defrag a re-numbering rather than a copy. The grid is, in this sense, the cache.

The next chapter turns to the way the grid is partitioned across sequences — the question of streams. It is the question of whether one big shared cache, or many small per-conversation caches, is the right model — and the answer is a flag with consequences.
