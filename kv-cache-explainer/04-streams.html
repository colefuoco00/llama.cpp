Streams and Sequences

One cache for everyone, or a cache per conversation. Two layouts, one flag, and the consequences of either choice.

A sequence is what the user has — a conversation. A stream is what the cache has — a slice of itself. The mapping between them is yours to pick.

The cache's third tensor dimension, mentioned and dismissed in the previous two chapters, is n_stream. Its purpose is partitioning. When n_stream is 1, the entire kv_size capacity is one large pool, and every sequence the cache holds competes for slots in that pool. When n_stream is greater than 1, the capacity is divided into per-stream sub-pools, and each sequence is assigned a stream at creation time. The mapping from sequence to stream is the seq_to_stream array.

If your model serves a single user conversation at a time, n_stream = 1 is almost certainly the right answer. The interesting case is the one where several conversations are in flight simultaneously and the question becomes: should they share, or be isolated?

Two regimes, side by side

Try toggling between them.

The unified case

In unified mode, n_stream is 1 and every sequence shares the cache's slots. The seq_to_stream mapping is uniformly zero. The advantages are real: sequences can be batched together in a single ubatch, the allocator has one pool to work with (no fragmentation between streams), and prefix-sharing between sequences is as cheap as setting an additional bit in a cell's seq bitset.

The cost is that a sequence which generates a great deal of context will eat into the capacity available to other sequences. If you have a fixed budget — say, eight users hitting a server, each capped at 4k tokens — unified mode does not by itself enforce that cap. The allocator simply hands out slots until it runs out. The cap has to be enforced by the caller, with seq_rm.

The per-sequence case

In per-sequence mode, n_stream equals the maximum number of sequences the cache will hold, and each sequence is bound to its own stream. The streams cannot exchange cells: sequence 3 cannot reach into stream 1 to find a slot. The cap is structural.

The advantages: per-sequence isolation, predictable memory per conversation, no contention between sequences for slots. The disadvantages: no prefix sharing (the cells are in different streams, so a fork must be a copy), more total memory (each stream is a separate pool), and reduced batching opportunities — since each sequence has its own cell layout, some operations have to be performed per-stream rather than across the whole cache.

Which to choose

If your sequences share prefixes (speculative decoding, chat-with-system-prompt, batch eval over the same context), unified. If your sequences are independent users on a shared server and you need a hard cap per user, per-sequence. The default in llama.cpp is unified — but a number of llama-server deployments turn it off for the isolation guarantees.

Stream copies, deferred

One subtlety. When a seq_cp operation copies a sequence's cached tokens to another sequence — say, forking a draft from the trunk — the destination might be in a different stream. The two streams are different physical regions of the K and V tensors; this is a real data copy on the device, not just a metadata flip.

The cache does not perform that copy immediately. It queues the request in a sc_info (stream-copy info) structure and defers it until the next update() call, where it executes as ggml_backend_tensor_copy on the device. The deferral lets the cache batch multiple copies, plan the allocator without committing to data motion, and recover gracefully if the operation is abandoned. The user-facing semantics are unchanged — after seq_cp returns, the destination behaves as if the copy happened — but the bytes haven't moved yet.

§ 4.4 — invariant Within a single stream, seq_cp is a metadata operation: the destination simply joins the source's bitset. Across streams, it is a deferred device copy. The caller never has to know which one happened.

A note on naming

The word stream is overloaded in any system that also runs on a GPU: CUDA streams, audio streams, IO streams. The streams discussed in this chapter are none of those. They are a logical partitioning of cache slots, internal to llama.cpp. The same code, on the same GPU, with the same number of CUDA streams, will behave differently depending on whether n_stream is 1 or greater. We mention this because it confuses the first reader of the source every time.

Having now described what cells are and how they are grouped into streams, we have most of the structural vocabulary the rest of the cache uses. One large piece remains before we can talk about operations on the cache: the variants. The cache we have been describing is the unified one. There is a second, the ISWA wrapper, which exists for models with sliding-window attention. The next chapter explains it.
