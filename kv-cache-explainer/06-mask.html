Building the Mask

A dense float matrix that decides, for every query token in the current ubatch, which past slots it is allowed to see.

The cache stores everything that could be attended to. The mask decides what will be. It is where causality, sequence isolation, and sliding windows are all enforced at once.

When an attention op runs, it does not consult the cells directly. It consults a mask — a dense matrix of additive bias values, one per (query token, key slot) pair. Where the mask is zero, the attention proceeds normally. Where the mask is -∞, the softmax sends the probability to zero and the slot is, for this query, invisible. Constructing this matrix correctly is the job of set_input_kq_mask, and it is the place where the cache's metadata becomes operationally consequential.

The shape of the matrix

The mask is dimensioned [n_kv, n_tokens]: for every cached slot the cache considers reachable, and every query token in the current ubatch, one float. In the unified case, n_kv is at most kv_size — the full cache capacity — though in practice the loop early-outs on slots that are empty. For each (q, k) cell of the matrix, the question is: does the query at position q attend to the key at slot k? The cache answers by walking a checklist.

conceptual — the checklistmask[q, k] = -∞   if any of:
                  cells[k].pos == -1            // the slot is empty
                  cells[k].seq.has(seq[q]) == false  // wrong sequence
                  cells[k].pos > pos[q]            // future (non-causal)
                  cells[k].pos < pos[q] - n_swa   // outside SWA window
              =  0    otherwise

Four conditions, in this order: emptiness, sequence membership, causality, and the sliding window. Each is a single check against the cell's metadata. Together they produce the mask. The implementation, of course, is more careful — it caches results across consecutive query tokens that share a sequence, it handles ALiBi position biases, it handles 2-D positions for M-RoPE models — but the conceptual content is the four-line predicate above.

Where it's built

The implementation lives in set_input_kq_mask, in src/llama-kv-cache.cpp. The body is a template parameterised on whether SWA is in effect, whether the model uses ALiBi, and whether 2-D positions are needed; the template lets the compiler emit a tight, branchless inner loop for each combination instead of leaving the predicate evaluation to runtime. The structure of one specialisation:

src/llama-kv-cache.cpp — ~ line 1418template <bool swa, bool alibi>
void set_input_kq_mask_impl(mask_args & args) {
    for (uint32_t q = 0; q < n_tokens; ++q) {
        const llama_pos    qp = ubatch.pos[q];
        const llama_seq_id qs = ubatch.seq_id[q];

        for (uint32_t k : cells.used) {
            const llama_pos kp = cells.pos[k];

            if (!cells.seq[k].has(qs))   { mask[q*n_kv + k] = -INFINITY; continue; }
            if (kp > qp)                  { mask[q*n_kv + k] = -INFINITY; continue; }
            if constexpr (swa) {
                if (kp < qp - int(n_swa)) { mask[q*n_kv + k] = -INFINITY; continue; }
            }

            mask[q*n_kv + k] = constexpr (alibi) ? alibi_bias(qp - kp) : 0.0f;
        }
    }
}

Note that the outer iteration is over cells.used, the derived set of non-empty slots. The empty slots have their mask entries pre-filled with -∞ elsewhere and are not revisited; the inner loop never has to handle them. This is what makes the cost of building the mask scale with the occupied portion of the cache rather than its total size.

A cache that helps

One observation worth surfacing. Two query tokens belonging to the same sequence and adjacent in position have almost identical masks: only the cells at the boundary of their respective causal frontiers differ. The implementation exploits this. For consecutive ubatch tokens with the same sequence, it copies the previous query's mask row and edits only the entries that have to change. Over a long ubatch, this saves a substantial fraction of the per-token work, especially for SWA configurations where the inner loop is tight.

§ 6.4 — perf A mask for a 32k context and a 256-token ubatch is an 8M-entry float array. Building it once is fine; building it from scratch for every layer would dominate the decode loop. So it is built once per forward pass and reused across all attention layers.

ALiBi and 2-D positions

The default mask entry is zero — the slot is admissible, with no additive bias. Two model families ask for more.

ALiBi models replace the zero with a small negative number proportional to the distance (qp - kp). The proportionality constant is per-head and is one of the model's hyperparameters. The effect is that attention falls off smoothly with distance, without an explicit window. The mask becomes a graded admissibility, not a binary one.

M-RoPE / 2-D positions models cache a (row, column) pair per cell, in the ext field we mentioned in Chapter III. The mask builder consults both coordinates when evaluating distance, and the SWA predicate becomes a 2-D box rather than a 1-D interval. This is rare in pure text models but central in some multimodal architectures, where the cache must represent image patches arranged in space as well as tokens arranged in time.

What the mask does not do

It is worth stating what the mask is not. It is not the only thing that limits which slots an attention can use: the cells.used set already excludes the empty ones, so the mask never needs to declare them inadmissible. It is not the place where eviction decisions are made: that happens elsewhere, by seq_rm or by SWA-aware allocation. And it is not where prefix sharing is implemented: shared cells appear in the mask once, with multiple sequences eligible to attend, and the mask predicate handles each sequence's perspective independently.

The mask is the moment, in each forward pass, when the cache's accumulated bookkeeping meets the math. Every careful invariant the cache has maintained — every shift, every fork, every sliding window — collapses into a single dense matrix of zeros and infinities. Then the softmax does the rest.

We now have the full vocabulary. The remaining two chapters describe the operations on the cache — the lifecycle of a forward pass and the seq_* family — and the recent additions worth knowing about: quantized KV, multi-token prediction, hybrid memory.
