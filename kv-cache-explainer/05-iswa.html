Two Caches, Interleaved

When some layers see the whole past and others only a window into it. The ISWA wrapper, Gemma, and the merits of asking less of a sliding cache.

A model that mixes full-attention and sliding-window layers has two different memory budgets. It deserves two different caches.

Some recent architectures — Gemma in particular — interleave two kinds of attention layers. The first kind is the familiar one: every query attends to every prior token. The second kind is sliding-window attention (SWA): every query attends only to the most recent W tokens, where W is the window size. The interleaving pattern is fixed per model: a Gemma block might be six SWA layers followed by one full layer, repeating; the full layers are sparser and do the long-range work, while the SWA layers handle the local context cheaply.

The cache implication is direct. The SWA layers only ever consult the last W tokens; their cache need only hold that many. The full layers consult the entire history; their cache must hold all of it. One single cache sized for the full layers would waste enormous quantities of memory on the SWA layers. One single cache sized for the SWA layers would not work at all for the full ones. The answer is two caches, composed.

The wrapper

This composition is the job of llama_kv_cache_iswa, defined in src/llama-kv-cache-iswa.{h,cpp}. It implements the same llama_memory_i interface as the unified cache — same methods, same semantics — but internally holds two underlying llama_kv_cache objects:

src/llama-kv-cache-iswa.h — line ~14class llama_kv_cache_iswa : public llama_memory_i {
    std::unique_ptr<llama_kv_cache> kv_base;  // for full-attention layers
    std::unique_ptr<llama_kv_cache> kv_swa;   // for SWA layers

    // every method on this class delegates to one or both of the above.
};

It is a strict delegation. Methods that affect cache state — seq_cp, seq_rm, seq_add, clear, defrag — run on both underlying caches in turn, because both caches own metadata for the same sequences. Methods that produce a graph or a slot allocation are dispatched to the appropriate underlying cache for each layer based on whether that layer is SWA or full.

Sizing the SWA cache

The savings come from the SWA cache's size. The constructor for llama_kv_cache_iswa computes it as:

src/llama-kv-cache-iswa.cpp — line ~46const uint32_t size_swa =
    std::min(size_base, hparams.n_swa * n_seq_max + n_ubatch);

// padded to a multiple of 256 for backend-friendly shapes
const uint32_t size_swa_padded =
    GGML_PAD(size_swa, 256);

Three terms, each justified. The n_swa is the window width, which is what any single sequence can possibly attend to in an SWA layer. We multiply by n_seq_max because the cache must accommodate that many sequences concurrently, each with its own window. We add n_ubatch as headroom: a single ubatch may temporarily extend a sequence's footprint beyond the window before the older tokens are evicted, and the allocator needs slack to land that ubatch somewhere. The min with size_base caps the SWA cache at the full-cache size, since there is no point making the supposedly-smaller cache larger than the larger one.

The pad to 256 is for the underlying tensor shapes: many GGML kernels prefer their second dimensions to be a multiple of 256, and the small over-allocation buys back any performance the awkward shape would have cost. A Gemma 4 model in this scheme might have size_base = 32k and size_swa ≈ 2.3k, depending on the window width — a fourteen-fold reduction in the cache memory needed for the SWA layers.

§ 5.2 — gemma Gemma 2 / 3 use a 4k SWA window with a much larger full-context budget. Gemma 4 keeps the same composition. The ISWA wrapper was added to llama.cpp specifically to give those models a tractable memory profile.

The delegation pattern

The wrapper's methods follow a single pattern: dispatch to both children, then combine. seq_rm(seq, p0, p1) removes the sequence from both caches; seq_cp copies it in both; clear() clears both. The pattern is mechanical and the file that implements it is correspondingly short:

src/llama-kv-cache-iswa.cpp — line ~75bool llama_kv_cache_iswa::seq_rm(llama_seq_id seq_id, llama_pos p0, llama_pos p1) {
    bool r = true;
    r &= kv_base->seq_rm(seq_id, p0, p1);
    r &= kv_swa->seq_rm(seq_id, p0, p1);
    return r;
}

void llama_kv_cache_iswa::clear() {
    kv_base->clear();
    kv_swa->clear();
}

// ...and so on for every method that mutates state.

The one place where the wrapper does more than delegate is when it produces a memory context for use by the model's forward pass. There, it has to know which layers belong to which cache, and it returns a composite context that routes layer-level queries to the right child. The model's graph-building code doesn't need to know which kind of cache it has; it asks for the K and V buffers for its current layer, and the right ones come back.

When ISWA is, and isn't

The wrapper exists only when the model has both kinds of layers. Models without SWA layers do not need it; models with only SWA layers (rare in practice) could use a single SWA-sized cache. The decision is made at load time based on the model's hyperparameters: if any layer has a non-zero window, and any layer doesn't, the cache is an llama_kv_cache_iswa; otherwise it's a plain llama_kv_cache.

For our purposes, the important takeaway is that the ISWA wrapper does not change how the cache works. It composes two caches that work the way the previous chapters have described. The only thing that changes is the routing: a query in an SWA layer consults the SWA cache, a query in a full layer consults the full one, and the wrapper makes that consultation happen automatically.

With that, we have described every container of cached state in llama.cpp. The next chapter steps inside attention to ask the question that turns the cache from a passive store into an active filter: how is the mask built? Which slot is admissible for which query? The answer is a dense float matrix and a careful loop, and we are now ready for it.
