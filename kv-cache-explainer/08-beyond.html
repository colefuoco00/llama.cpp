Frontiers

Quantization & Beyond

Storing K and V in eight bits. Sharing the cache with multi-token-prediction heads. Hybrid memory for models that aren't just attention.

A cache that fits in memory is a cache that gets used. Two extensions have moved the cache toward that goal; a third has begun to ask whether the cache is the only kind of memory worth having.

The seven previous chapters described a cache of full-precision tensors, with metadata for one kind of model: a stack of attention layers, all producing K and V of similar shape. The cache supports more than that. This chapter rounds out the picture with three additions, each of which extends the cache without violating its core abstractions: quantized storage, multi-token-prediction support, and hybrid recurrent-attention memory.

Quantized KV

We previewed this in Chapter II: the constructor accepts type_k and type_v independently, and any GGML quantization type can be specified. The two are not symmetric in practice.

K is the more forgiving target. The dot products Q·Kᵀ are normalised by √d and passed through a softmax, both of which absorb small numerical errors. Quantizing K to q8_0 is, in practice, lossless for most models; quantizing to q4_0 is noticeable but tolerable. V is the more demanding target — the post-softmax values are mixed with V vectors directly, and errors there propagate unmasked. q8_0 V is fine; q4_0 V tends to degrade quality more visibly. The typical configuration in serving deployments is q8_0 for both, with q4_0 K and q8_0 V as a more aggressive option.

The implementation cost of all this is zero, in the sense that the cache code does not change between configurations. The tensor is allocated with the requested type, GGML's ggml_mul_mat handles the dequantization in its kernel, and the seq_* operations don't read the cells' bytes at all. From the cache's point of view, what's in those bytes is opaque — which is exactly the property that makes quantization a knob the user can turn rather than a feature that has to be implemented.

§ 8.1 — note The defrag and stream-copy operations use ggml_backend_tensor_copy, which is layout-aware and dtype-aware. Copying a q8_0 slot to another q8_0 slot moves the quantized bytes directly, without dequantizing in between.

Multi-token prediction

Gemma 4 introduces multi-token-prediction (MTP) heads: small additional decoder stacks that share the trunk model's hidden states up to some layer and then branch off to produce a draft prediction for the next token alongside the main model's prediction for the current one. The MTP heads run alongside the main forward pass; their drafts can be verified by the trunk on the following step, giving speculative-decoding-like speedups without a separate draft model.

From the cache's point of view, MTP heads pose a single question: do they need their own K and V buffers, or do they share the trunk's? The answer depends on the layer. For layers the head re-uses from the trunk, the K and V are shared — there is no separate cache. For layers the head introduces, the K and V are new and need to be stored somewhere. The cache supports this with a per-layer flag, kv_only_nextn, that tells the cache to allocate buffers only for the final n layers (those introduced by the MTP head).

The MTP head also wants to attach its own forward pass to the trunk's existing context — it shouldn't have to re-tokenize and re-cache the prefix. So the cache exposes seq_pos_max(seq_id) on its memory context: a way for the head to ask the trunk's cache "what's the highest cached position for this sequence?" and to anchor its draft predictions accordingly. The head's cache is small (only the layers it owns) but reads from the trunk's positions seamlessly.

Hybrid memory: not just attention

Some modern architectures mix attention layers with recurrent layers — Mamba blocks, RWKV variants, or other state-space models. Recurrent layers do not have a KV cache; they have a small, fixed-size recurrent state that evolves with each token. The cache abstraction doesn't fit. The solution: a hybrid wrapper.

llama_memory_hybrid, in src/llama-memory-hybrid.{h,cpp}, composes a llama_kv_cache for the attention layers with a recurrent-state buffer for the recurrent ones. Like the ISWA wrapper of Chapter V, it implements the llama_memory_i interface and delegates to its children. Unlike the ISWA wrapper, its two children are different kinds of memory — a cache and a recurrent state — with different semantics. The seq_* operations dispatch to both, and the graph-building code routes each layer's query to the appropriate buffer.

The hybrid wrapper is a recent addition and is exercised by a handful of architectures so far — RWKV variants and Mamba-style hybrids. The KV cache itself, viewed from outside the wrapper, is unchanged; it manages its layers exactly as before. But the abstraction it sits behind has been generalized: a model's memory is now a polymorphic llama_memory_i, not specifically a KV cache. That generalization is what made it possible to add hybrid models without changing every caller.

A few miscellaneous corners

Three smaller details, mentioned for completeness.

Variable V-cache padding. As described in Chapter II, when V widths differ per layer and flash attention is not in use, the cache pads all V buffers to the maximum width. The flag TAG_V_CACHE_VARIABLE records this. With flash attention, no padding is needed; the kernel handles variable widths directly.

get_can_shift(). Not every model can have its cache shifted: some position-encoding schemes don't support it, or do so only for certain ranges. The cache offers a method that returns whether a shift would be safe. The seq_add and seq_div operations call it before queueing their work; if it returns false, they return an error rather than corrupt the cache.

The memory snapshot API. The library exposes a way to serialize the cache state to bytes and load it back. This is what makes "save a session and resume it later" possible — the cache, including all its metadata and data, is a first-class persistable object. The serialization respects quantization: a quantized cache loads back in its original dtype, no reconversion required.

A closing observation

What strikes me, having walked the cache end to end, is how much of its complexity has been absorbed into metadata. Forking a sequence: a bit in a set. Shifting a window: an integer in an array, applied later by a graph. Composing two attention regimes: a wrapper that delegates. Quantization: a tensor dtype, transparent to the cache logic. Whenever an interesting feature has been added, the cache has answered it by adding to its metadata or by composing existing pieces, not by rewriting the data path.

The data path — the K and V tensors and the matmuls that consume them — has remained almost unchanged across years of additions. This is, I think, the right outcome for a cache. The data is what's expensive; the metadata is what's flexible. Llama.cpp's KV cache draws the line between them with care, and most of the cache's surface area is on the metadata side, where care is cheap.
